## 1. Overview

The Evidence Log is the authoritative system of record for all execution evidence generated by Daedalus. It is not a running service. It is a specification of persistence semantics, identity, lifecycle, and retrieval requirements for the artifacts produced by a single job execution.

The Evidence Log enables:

- **Reproducibility** — Any execution can be re-run from stored state given the reproducibility tuple.
- **Benchmarking** — Solver performance across executions can be compared using stored metrics.
- **Evaluation** — Quality evaluation results are preserved alongside the solver output that produced them.
- **Auditability** — The complete record of a scheduling decision is queryable after the fact.
- **Experiment comparison** — Executions of the same problem under different backends or parameters are comparable.
- **Scheduler effectiveness analysis** — Predicted outcomes can be compared against actual outcomes to evaluate scheduler decisions.

The persistence backend is PostgreSQL. The component responsible for writing evidence is the Worker (SPEC-005). The API layer is responsible for creating the initial job record. No other component may write to Evidence Log artifact tables.

## 2. Glossary

| Term | Definition |
|---|---|
| **Evidence record set** | The complete collection of artifacts stored for one `job_id`. |
| **Job record** | The lifecycle state record for a job. Created by the API; updated by the Worker. |
| **Decision record** | The scheduler decision record produced by Core and persisted by the Worker. Defined by SPEC-003 FR-10. |
| **Solver run record** | The artifact capturing solver execution inputs and outputs, including the reproducibility tuple. |
| **Quality evaluation record** | The artifact capturing the quality evaluation result for a solver run. Schema defined by the Core Quality Evaluation Specification. |
| **Failure record** | The artifact capturing structured failure detail for a job that failed before or during solver invocation. |
| **Report metadata record** | The artifact capturing metadata about a generated evidence report. Minimum fields defined here; schema extended by the Report Generator Specification. |
| **Reproducibility tuple** | The four-value set `(problem_id, execution_seed, backend_id, contract_version)` sufficient to reproduce a solver execution. |
| **`job_id`** | The immutable, system-assigned identifier for a job. The idempotency key for all Evidence Log upsert operations. |

## 3. Functional Requirements

### FR-1: Responsibility Scope and Component Boundaries

**3.1.1** The Evidence Log owns the following responsibilities:
- Evidence persistence semantics for all artifact types defined in this specification.
- The identity model for evidence records (`job_id` as the root identifier).
- The lifecycle of evidence from creation through terminal state.
- Idempotent persistence behavior for all artifact types.
- Retention requirements for stored evidence.
- Retrieval requirements sufficient to support the use cases enumerated in Section 1.

**3.1.2** The Evidence Log does not own the following responsibilities:
- The routing problem itself. The routing problem is an immutable artifact owned by SPEC-001 and referenced by `problem_id`. The Evidence Log references the routing problem by `problem_id`; it does not duplicate routing problem data.
- The decision logic that produced the decision record. Decision logic is owned by SPEC-003.
- The solver contract or solver execution semantics. Those are owned by SPEC-004 and SPEC-005.
- The quality evaluation algorithm or quality evaluation schema. Those are owned by the Core Quality Evaluation Specification (pending).
- The report generation algorithm or report schema. Those are owned by the Report Generator Specification (pending).
- Job scheduling, message queue management, or Worker lifecycle.

**3.1.3** No component other than the Worker (SPEC-005) may write to Evidence Log artifact tables after job creation. The API layer creates the initial job record. The Worker creates and updates all remaining artifact records.

---

### FR-2: Evidence Identity and Record Set Model

**3.2.1** Each job is identified by a `job_id`. The `job_id` is the root identifier for the complete evidence record set associated with that job.

**3.2.2** A complete evidence record set for a job in `Completed` state consists of:
1. One job record with `status = Completed`.
2. One decision record.
3. One solver run record.
4. Zero or one quality evaluation records (present when quality evaluation was invoked; may be absent when quality evaluation failed or was skipped — see FR-7).
5. Zero or one report metadata records (present only when a report has been generated — see FR-9).

A `Completed` job does not have a failure record. Failure records are present only for `Failed` jobs (see FR-8.5 and FR-13.3).

**3.2.2a** A complete evidence record set for a job in `Failed` state consists of:
1. One job record with `status = Failed`.
2. One failure record. SPEC-005 FR-13 guarantees that a failure record is persisted before the NACK is issued for all permanent failures that transition the job to `Failed` state.
3. Zero or one decision records (present when the failure stage reached Scheduling (SPEC-005 FR-5) or later; absent when the failure stage was Load (SPEC-005 FR-4) or earlier).

**3.2.3** A `job_id` uniquely identifies one evidence record set. No two jobs share a `job_id`.

**3.2.4** The routing problem is referenced by `problem_id` in the solver run record and the decision record. The problem data is not stored in the Evidence Log.

**3.2.5** The reproducibility tuple `(problem_id, execution_seed, backend_id, contract_version)` fully characterizes the inputs to a solver execution, sufficient for reproduction. The solver run record must store all four values. The tuple identifies the execution configuration; `job_id` identifies the specific job instance. Two separate jobs with identical reproducibility tuples will produce identical solver outputs when both yield `Succeeded`, per SPEC-004 FR-11's reproducibility invariant.

---

### FR-3: Evidence Lifecycle

**3.3.1** A job record enters the Evidence Log when the API layer creates it in the `Pending` state.

**3.3.2** The job lifecycle states — `Pending`, `Processing`, `Completed`, and `Failed` — are defined by SPEC-005 FR-2, which is the authoritative definition. The Evidence Log stores the current lifecycle state in the job record (FR-4). The Worker is the authoritative writer of lifecycle state after the API creates the job in `Pending` state.

**3.3.3** State transitions are monotonic. A job that enters `Processing` state does not revert to `Pending` on re-delivery; it is re-processed in `Processing` state via idempotent operations (SPEC-005 FR-14). Once a job reaches `Completed` or `Failed`, it does not transition again.

**3.3.4** A job in `Completed` or `Failed` state is terminal. The Worker must not modify the job state after reaching a terminal state. On message re-delivery for a terminal-state job, the Worker discards the message and ACKs without modifying any evidence artifact (SPEC-005 FR-14).

**3.3.5** The Evidence Log does not specify timeouts or expiration policies that would cause an in-progress job to be marked `Failed` without Worker action. Stale `Processing` state cleanup is an implementation planning concern.

---

### FR-4: Job Record Persistence

**3.4.1** The job record captures the lifecycle state of a job. It is the root record for the evidence record set.

**3.4.2** The job record must store:

| Field | Type | Constraints |
|---|---|---|
| `job_id` | string | Immutable; system-assigned; unique; primary key |
| `problem_id` | string | Immutable; set at creation; references SPEC-001 problem identity |
| `status` | enum | `Pending`, `Processing`, `Completed`, `Failed` |
| `created_at` | timestamp | Immutable; set by API at job creation; UTC |
| `updated_at` | timestamp | Updated on every state transition; UTC |
| `completed_at` | timestamp | Set when status transitions to `Completed`; null otherwise |
| `failed_at` | timestamp | Set when status transitions to `Failed`; null otherwise |

**3.4.3** The job record must support `job_id`-keyed upsert for all fields mutable after creation. `job_id` and `created_at` are immutable after creation.

**3.4.4** The Worker sets `status` to `Processing` when it begins execution. The Worker sets `status` to `Completed` or `Failed` when the job reaches a terminal state.

---

### FR-5: Decision Record Persistence

**3.5.1** The decision record is produced by Core (the Scheduler, SPEC-003) and persisted by the Worker. The Worker must persist the decision record as received, without modification, after receiving it from Core.

**3.5.2** The decision record schema is defined by SPEC-003 FR-10. The Evidence Log adopts that schema in full. The fields are reproduced here for traceability; SPEC-003 FR-10 is authoritative:

| Field | Source |
|---|---|
| `decision_id` | SPEC-003 FR-10 |
| `problem_id` | SPEC-003 FR-10 |
| `objective_mode` | SPEC-003 FR-10 |
| `workload_features_snapshot` | SPEC-003 FR-10 |
| `selected_backend_id` | SPEC-003 FR-10 |
| `candidate_scores` | SPEC-003 FR-10 |
| `rejection_reasons` | SPEC-003 FR-10 |
| `predicted_latency` | SPEC-003 FR-10 |
| `predicted_cost` | SPEC-003 FR-10 |
| `predicted_quality` | SPEC-003 FR-10 |
| `confidence_score` | SPEC-003 FR-10 |
| `decision_status` | SPEC-003 FR-10 |
| `timestamp` | SPEC-003 FR-10 |
| `actual_outcome` | SPEC-003 FR-10 (reserved; populated post-execution) |
| `hindsight_quality` | SPEC-003 FR-10 (reserved; populated post-quality-evaluation) |

**3.5.3** The Evidence Log stores the decision record keyed by `job_id`. The `decision_id` is a correlation identifier within a single execution attempt. On re-execution (message re-delivery), the Worker receives a new `decision_id` from Core. The new decision record replaces the prior attempt's record via `job_id`-keyed upsert.

**3.5.4** The `actual_outcome` field in the decision record is updated by the Worker after the solver run completes, using the `SolverOutcome` value from the SolverResponse (SPEC-004 FR-4). The `hindsight_quality` field is updated by the Worker after quality evaluation completes, if quality evaluation succeeds.

**3.5.5** The decision record must support `job_id`-keyed upsert. A second write for the same `job_id` updates the existing record; it does not create a duplicate.

---

### FR-6: Solver Run Record Persistence

**3.6.1** The solver run record captures the complete inputs and outputs of a solver execution, including the reproducibility tuple.

**3.6.2** The solver run record must store the following fields:

| Field | Source | Notes |
|---|---|---|
| `job_id` | Worker | Foreign key to job record |
| `problem_id` | SolverRequest (SPEC-004 FR-2) | Part of reproducibility tuple |
| `execution_seed` | Worker (SPEC-005 FR-7, ADR-010) | Part of reproducibility tuple; derived by Worker using exclusive seed authority |
| `backend_id` | Worker / Scheduler decision | Part of reproducibility tuple |
| `contract_version` | Worker (SPEC-005) | Part of reproducibility tuple; the `contract_version` value from the SolverRequest (the Worker's intended contract version at dispatch time — see FR-6.2a) |
| `solver_outcome` | SolverResponse (SPEC-004 FR-3) | `SolverOutcome` enum value (SPEC-004 FR-4) |
| `route_plan` | SolverResponse (SPEC-004 FR-3) | Present when the SolverResponse included a solution: required when `solver_outcome = Succeeded`; optional when `solver_outcome = Timeout` or `Cancelled` (SPEC-004 FR-9, FR-10). Null when no solution was returned (`Infeasible`, `Failed`, or `Timeout`/`Cancelled` with no solution present). |
| `execution_statistics` | SolverResponse (SPEC-004 FR-3) | `ExecutionStatistics` (SPEC-004 FR-6); null when absent |
| `failure_detail` | SolverResponse (SPEC-004 FR-3) | `SolverFailureDetail` (SPEC-004 FR-8); null when outcome is `Succeeded` |
| `extension_metadata` | SolverResponse (SPEC-004 FR-13) | Backend-specific metadata; null when absent |
| `structural_validation_status` | Worker | Whether the RoutePlan passed the Worker's structural validation check (SPEC-005 FR-11) |
| `persisted_at` | Worker | Timestamp at which the Worker persisted this record; UTC |

**3.6.2a** The `contract_version` stored in the solver run record is the value from the SolverRequest — the Worker's intended contract version at the time of dispatch. Under a `ContractVersionMismatch` failure (SPEC-004 FR-14), this is the version the Worker sent; the backend's declared version is diagnostic context available in the `failure_detail` field.

**3.6.3** The reproducibility tuple `(problem_id, execution_seed, backend_id, contract_version)` must be complete in every solver run record. A solver run record missing any element of the reproducibility tuple is a persistence error.

**3.6.4** The `execution_seed` field must not be logged in structured log events (SPEC-005 observability requirements). It is stored exclusively in the solver run record.

**3.6.5** The solver run record must support `job_id`-keyed upsert. A re-execution produces a new solver run record that replaces the prior attempt's record via `job_id`-keyed upsert.

---

### FR-7: Quality Evaluation Record Persistence

**3.7.1** The quality evaluation record captures the result of the Core quality evaluation invoked after solver execution (SPEC-005 FR-15).

**3.7.2** The quality evaluation record schema is defined by the Core Quality Evaluation Specification (pending). That specification is authoritative for the quality evaluation record schema. The Evidence Log does not define quality evaluation schema fields except as follows.

**3.7.3** The quality evaluation record must include at minimum:

| Field | Notes |
|---|---|
| `job_id` | Foreign key to job record |
| `evaluation_status` | Whether evaluation completed, was skipped, or failed |
| `evaluated_at` | Timestamp of evaluation; UTC; null when evaluation absent |

**3.7.4** When Core quality evaluation fails due to an infrastructure failure (SPEC-005 FR-15 failure handling), the quality evaluation record is persisted with `evaluation_status` set to `Failed` and all quality score fields null. The absence of a quality evaluation result must not prevent the job from reaching `Completed` state.

**3.7.5** When quality evaluation succeeds, the Worker updates the `hindsight_quality` field in the decision record via `job_id`-keyed upsert, using the quality score returned by Core quality evaluation (SPEC-005 FR-15).

**3.7.6** The quality evaluation record must support `job_id`-keyed upsert.

**Open Question OQ-1 (Blocking):** The Core Quality Evaluation Specification has not been authored. The quality evaluation record schema is deferred until that specification is accepted. SPEC-006 FR-7 cannot advance to Accepted until OQ-1 is resolved.

---

### FR-8: Failure Record Persistence

**3.8.1** The failure record captures structured failure detail for a job that failed before or during solver invocation. It does not capture solver-level failure outcomes; those are captured in the solver run record via `failure_detail` (FR-6).

**3.8.2** The failure record is persisted when a job fails due to:
- A missing or unresolvable routing problem (problem load failure).
- A Core contract violation (unexpected response type from Core).
- Scheduler rejection (Core returns a rejection decision rather than a selected backend).
- Solver invocation failure before any SolverResponse is received.

When the failure stage is `Schedule` (NoEligibleSolver, InvalidConfiguration, or other Scheduler decision failures in SPEC-005 FR-5), both a decision record and a failure record are persisted. The decision record carries the full per-backend rejection detail (SPEC-003 FR-9, FR-10), which is the authoritative record of why no solver was selected. The failure record provides a uniform failure stage classification queryable across all failure types without joining to the decision record. Both records serve distinct query purposes and together constitute the complete failure evidence for Schedule-stage failures.

**3.8.3** The failure record must store:

| Field | Notes |
|---|---|
| `job_id` | Foreign key to job record |
| `failure_stage` | The stage at which the failure occurred: `Load`, `Schedule`, `Invocation` |
| `failure_class` | Structured failure classification consistent with SPEC-005 failure taxonomy |
| `failure_detail` | Human-readable description; must not include execution_seed |
| `is_retryable` | Whether the failure is classified as transient (NACK+requeue) or permanent (dead-letter) |
| `failed_at` | Timestamp; UTC |

`is_retryable` records the failure classification per SPEC-005 FR-13 at the time the failure record is written. For jobs that reached `Failed` via transient-failure retry exhaustion, `is_retryable` reflects the original transient failure classification (the failure was retryable, but retries were exhausted by the broker). For immediately permanent failures, `is_retryable` is `false`. This field supports diagnostic queries distinguishing "immediately permanent" from "transient but retry-limit-exhausted" failure histories.

**3.8.4** The failure record must support `job_id`-keyed upsert.

**3.8.5** A job that reached `Completed` state does not have a failure record, even if quality evaluation failed. Quality evaluation failure is captured in the quality evaluation record (FR-7.4), not the failure record.

---

### FR-9: Report Metadata Record Persistence

**3.9.1** The report metadata record captures metadata about a generated evidence report. It is created by the Worker after successful report generation (SPEC-005 FR-17).

**3.9.2** The report metadata record minimum fields are:

| Field | Notes |
|---|---|
| `job_id` | Foreign key to job record |
| `report_id` | System-assigned identifier for the report artifact |
| `report_format` | The format of the generated report (e.g., `json`, `html`) |
| `generated_at` | Timestamp of report generation; UTC |
| `generation_status` | Whether generation completed or failed |

**3.9.3** The Report Generator Specification (pending) may extend the report metadata record schema with additional fields. The fields defined in FR-9.2 are the minimum required at Evidence Log specification scope.

**3.9.4** The report metadata record must support `job_id`-keyed upsert.

**3.9.5** The absence of a report metadata record does not prevent a job from reaching `Completed` state. Report generation failure is captured in the Worker telemetry (SPEC-005 `worker.report.generation.failed` event); the job completes without a report metadata record.

**Open Question OQ-2 (Non-Blocking):** The Report Generator Specification has not been authored. The report metadata record schema extension is deferred until that specification is accepted.

### FR-10: Idempotency Requirements

**3.10.1** All Evidence Log artifact writes performed by the Worker must be idempotent. A second write of the same artifact for the same `job_id` must produce no duplicate records and must not fail.

**3.10.2** `job_id` is the natural idempotency key for all Evidence Log persistence operations. The idempotency mechanism is `job_id`-keyed upsert for all artifact tables.

**3.10.3** Idempotency is required because the Worker operates under at-least-once delivery semantics (SPEC-005). A message may be redelivered after a Worker crash, and the Worker must be able to re-execute without corrupting the evidence record set.

**3.10.4** On message re-delivery, the Worker re-executes the job from the beginning. Artifacts written by the prior attempt are overwritten via `job_id`-keyed upsert. The Worker does not checkpoint intermediate state.

**3.10.5** All artifact tables defined in this specification must support `job_id`-keyed upsert semantics. The specific upsert mechanism (e.g., `ON CONFLICT DO UPDATE` or equivalent) is an implementation planning concern. The behavioral requirement is that a second write for the same `job_id` produces no duplicate records.

---

### FR-11: `job_id` Ownership and Uniqueness

**3.11.1** `job_id` values are system-assigned by the API layer at job creation. No component other than the API layer may assign `job_id` values.

**3.11.2** `job_id` values are globally unique within the Daedalus system. No two jobs may share a `job_id`.

**3.11.3** `job_id` values are immutable after assignment. A `job_id` is never reused, even after a job reaches a terminal `Failed` state.

**3.11.4** The Worker receives the `job_id` from the job message consumed from the queue. The Worker does not generate `job_id` values.

---

### FR-12: Upsert Behavior

**3.12.1** An upsert operation on an Evidence Log artifact must:
1. Insert the artifact record if no record exists for the given `job_id`.
2. Update the existing artifact record in place if a record already exists for the given `job_id`.
3. Return successfully in both cases.
4. Never create a second record for the same `job_id`.

**3.12.2** For the decision record specifically: a re-execution on message redelivery may produce a new `decision_id` from Core. The new decision record replaces the prior attempt's decision record via `job_id`-keyed upsert. The `decision_id` is a correlation identifier within a single execution attempt, not a stable identifier across re-executions.

**3.12.3** For the solver run record specifically: a re-execution may produce a new `execution_seed`. The new solver run record replaces the prior attempt's solver run record via `job_id`-keyed upsert. The prior execution's seed is not retained after overwrite.

**3.12.4** The upsert strategy must be atomic. Partial writes — where some fields are updated and others are not — are not acceptable.

### FR-13: Evidence Completeness

**3.13.1** A job in `Completed` state must have the following artifacts present in the Evidence Log:
- One job record with `status = Completed`.
- One decision record. The `actual_outcome` field is populated when Core quality evaluation succeeded; it is null when quality evaluation failed (SPEC-005 FR-15 failure handling) or was not invoked (no RoutePlan present, per SPEC-005 FR-15 invocation rules).
- One solver run record with the complete reproducibility tuple.

**3.13.2** Quality evaluation record and report metadata record are not required for `Completed` state. Their absence is permitted and must not prevent the `Completed` transition.

**3.13.3** A job in `Failed` state must have the following artifacts present:
- One job record with `status = Failed`.
- One failure record (if the failure occurred before or during solver invocation).

**3.13.4** The Worker is responsible for ensuring evidence completeness before transitioning the job to a terminal state. The Worker persists all applicable artifacts before writing the terminal job status.

---

### FR-14: Evidence Retrieval Requirements

**3.14.1** The Evidence Log must support retrieval of the complete evidence record set for a given `job_id`. All artifact records associated with a `job_id` must be retrievable together.

**3.14.2** The Evidence Log must support retrieval of all evidence record sets for a given `problem_id`. This supports experiment comparison and benchmarking across executions of the same problem.

**3.14.3** The Evidence Log must support retrieval of all evidence record sets for a given `backend_id`. This supports benchmarking across solver backends.

**3.14.4** The Evidence Log must support retrieval of the reproducibility tuple for a given `job_id`. The `(problem_id, execution_seed, backend_id, contract_version)` values must be retrievable from the solver run record.

**3.14.5** The Evidence Log must support retrieval of decision records with `actual_outcome` present, filtered by `problem_id`. This supports scheduler effectiveness analysis.

**3.14.6** Retrieval interface design (API shape, query parameters, response schema) is an implementation planning concern. The requirements above specify what must be retrievable; they do not specify how the retrieval is exposed.

---

### FR-15: Retention Requirements

**3.15.1** Evidence records must be retained for at least the configured minimum retention period after the job's terminal timestamp (`completed_at` or `failed_at`). The minimum retention period is configurable at deployment time. The default minimum retention period is 90 days; this default requires Project Owner confirmation before SPEC-006 advances to Accepted (see OQ-3).

**3.15.2** Evidence records must not be automatically deleted before the minimum retention period expires.

**3.15.3** Evidence record deletion after the minimum retention period is an implementation planning concern. This specification does not define automated deletion behavior.

**3.15.4** The routing problem referenced by `problem_id` has independent retention semantics defined by SPEC-001. Evidence Log retention does not imply routing problem retention.

**3.15.5** After evidence records are deleted following the retention period, the reproducibility guarantee of FR-16 can no longer be satisfied for those executions. The routing problem persists independently (SPEC-001), but the execution artifacts containing `execution_seed`, `backend_id`, and `contract_version` are gone. The reproducibility guarantee applies within the evidence retention period only. Operators requiring long-term reproducibility must configure a retention period sufficient for their archival needs, or archive solver run records independently before deletion.

---

### FR-16: Reproducibility Requirements

**3.16.1** Given the reproducibility tuple `(problem_id, execution_seed, backend_id, contract_version)` retrieved from a solver run record, an operator must be able to reproduce the solver execution. This guarantee applies within the evidence retention period defined in FR-15; evidence records deleted after the retention period are no longer available for reproduction.

**3.16.2** The Evidence Log must store all four elements of the reproducibility tuple in the solver run record (FR-6.3). No element of the reproducibility tuple may be absent from a completed solver run record.

**3.16.3** The routing problem identified by `problem_id` is the canonical problem input for reproduction. SPEC-001 guarantees that routing problems are immutable — the problem identified by `problem_id` is identical at reproduction time to what it was at execution time.

**3.16.4** The `execution_seed` stored in the solver run record is the seed used by the Worker's PRNG derivation (SPEC-005 FR-7, ADR-010). Reproducing the seed derivation formula against the same inputs must produce the same `execution_seed`. The seed derivation formula is owned by the Worker specification.

**3.16.5** The `contract_version` stored in the solver run record identifies the solver contract version under which the execution occurred. Reproduction requires the same contract version to be available.

**3.16.6** `SPEC-001 FR-6` establishes that the routing problem contains a `seed` field. The problem's `seed` contributes to the `execution_seed` derivation but is not separately stored in the solver run record — it is recoverable from the routing problem via `problem_id`. The Evidence Log does not duplicate problem fields.

### FR-17: Auditability Requirements

**3.17.1** The complete execution history for a job must be reconstructable from the evidence record set. An auditor must be able to determine:
- When the job was created and by what input.
- Which backend was selected and why (from the decision record).
- What the solver produced.
- Whether quality evaluation was performed and what it found.
- Whether the job failed and at what stage.

**3.17.2** The decision record (FR-5) provides the complete scheduler decision including `objective_mode`, `candidate_scores`, `rejection_reasons`, `predicted_latency`, `predicted_cost`, `predicted_quality`, and `confidence_score`. This satisfies the "why was this backend selected" audit requirement.

**3.17.3** The `actual_outcome` and `hindsight_quality` fields in the decision record (updated post-execution and post-evaluation respectively) provide the comparative record for scheduler effectiveness analysis.

**3.17.4** Evidence records must not be modified after a job reaches a terminal state, except as required by `job_id`-keyed upsert on re-delivery (which is not possible after a terminal state is reached — see FR-3.4). Audit integrity depends on terminal-state record immutability.

**3.17.5** Timestamps must be stored in UTC for all evidence artifact records.

---

### FR-18: Telemetry

**3.18.1** Evidence Log persistence operations must be observable through the Worker's telemetry system. The following observability requirements must be satisfied:

1. **Evidence persistence event** — A structured log event must be emitted after each evidence persistence operation, capturing at minimum: the job identifier, the list of artifact types successfully written, and the duration of the persistence operation.

2. **Upsert collision detection event** — A structured log event must be emitted when a `job_id`-keyed upsert updates an existing artifact record (re-execution detected). The event must identify the job identifier and the artifact type affected.

3. **Persistence failure event** — A structured log event must be emitted when any artifact persistence operation fails. The event must identify the job identifier, the artifact type that failed, and the failure classification.

**3.18.2** The `execution_seed` must not appear in any structured log event, including evidence persistence events.

**3.18.3** Evidence Log persistence must be represented by an OpenTelemetry span emitted by the Worker as a child of the `job.consume` root span (SPEC-005 FR-19). The span must capture the artifact types written and reflect the success or failure of the persistence operation in its span status.

**3.18.4** The field schemas for all structured log events defined in FR-18.1 and the span definition for FR-18.3 are specified in SPEC-005 (FR-19 and Section 9), which is the authoritative owner of Worker lifecycle telemetry. SPEC-005 must define these telemetry artifacts consistently with the requirements stated in this section.

**3.18.5** Evidence Log persistence failures are classified as infrastructure failures per SPEC-005's transient/permanent failure taxonomy (SPEC-005 FR-13). PostgreSQL unavailability during persistence is a transient failure; the Worker NACKs with requeue.

---

### FR-19: Failure Handling

**3.19.1** Failure handling for Evidence Log persistence failures is owned by SPEC-005. This specification defines the persistence semantics; SPEC-005 defines the Worker's response to persistence failures.

**3.19.2** PostgreSQL unavailability during Evidence Log persistence is a transient failure. The Worker must NACK with requeue. The job record state must not be updated to `Failed` on transient persistence failure.

**3.19.3** A persistence failure that prevents writing the terminal job status (`Completed` or `Failed`) must result in the Worker NACKing with requeue. The message will be redelivered; the re-execution will attempt persistence again.

**3.19.4** The Evidence Log does not define retry behavior internal to a single execution attempt. Retry policy for PostgreSQL operations is an implementation planning concern.

**3.19.5** On re-execution after a transient persistence failure, all artifact writes are repeated via `job_id`-keyed upsert (FR-10). Partial artifact sets from prior failed attempts are overwritten.

---

## 4. Non-Requirements

The following are explicitly out of scope for this specification:

- Database schema definitions, table structures, index definitions, or column types. These are implementation planning concerns.
- Query performance requirements or index strategies.
- Data migration or schema versioning.
- Backup or disaster recovery policies.
- Access control or authorization for Evidence Log reads.
- Multi-tenancy.
- Export or archival formats.
- Streaming or real-time Evidence Log subscriptions.
- Stale `Processing` state cleanup for orphaned jobs.

---

## 5. Assumptions

1. PostgreSQL is the Evidence Log persistence backend. No other persistence backend is in scope for MVP.
2. The Worker is the sole writer of Evidence Log artifacts after the API creates the job record.
3. `job_id` values assigned by the API are globally unique and not reused.
4. The routing problem identified by `problem_id` is immutable (SPEC-001). Evidence Log reproducibility depends on this guarantee.
5. Quality evaluation is deterministic — given the same RoutePlan and routing problem, Core produces identical quality evaluation results on every invocation (SPEC-005 Assumption 9). This property must be established by the Core Quality Evaluation Specification.
6. At-least-once delivery semantics for job messages are a system invariant. All Evidence Log persistence operations must be idempotent by design.
7. A re-executed job may produce a different solver *outcome* than the prior attempt (e.g., `Timeout` instead of `Succeeded`) due to timing differences or external signal differences. When both executions produce `Succeeded`, SPEC-004 FR-11's reproducibility invariant guarantees identical solutions given identical inputs. The upsert model treats the most recent execution's artifacts as authoritative regardless of whether outcomes are identical.

---

## 6. Constraints

1. All Evidence Log artifact tables must support `job_id`-keyed upsert semantics. Tables that support insert-only semantics are architecturally incompatible with this specification's idempotency requirements.
2. The `execution_seed` must be stored exclusively in the solver run record. It must not appear in log events, audit trails, or any other artifact.
3. Timestamps must be stored in UTC.
4. Terminal-state evidence records must not be modified after a job reaches `Completed` or `Failed`.
5. The Evidence Log specification does not define database schemas. Any constraint requiring a specific schema representation is out of scope unless the representation has externally observable behavioral consequences.

---

## 7. Acceptance Criteria

- **AC-1**: Given a completed job, all required artifacts (job record, decision record, solver run record) are present and queryable by `job_id`.
- **AC-2**: Given a completed job, the reproducibility tuple `(problem_id, execution_seed, backend_id, contract_version)` is present and complete in the solver run record.
- **AC-3**: Given a re-delivered job message for a job that has already completed (terminal state), the Worker ACKs the message and no Evidence Log artifact is modified.
- **AC-4**: Given a re-delivered job message for a job that has not yet reached terminal state, the Worker re-executes and overwrites prior-attempt artifacts via `job_id`-keyed upsert. No duplicate artifact records are created.
- **AC-5**: Given a PostgreSQL failure during artifact persistence, the Worker NACKs with requeue and the job status is not updated to `Failed`.
- **AC-6**: Given a completed job where quality evaluation failed, the job record status is `Completed`, the quality evaluation record has `evaluation_status = Failed`, and no failure record is present.
- **AC-7**: Given a failed job (pre-solver failure), a failure record is present with `failure_stage` and `failure_class` populated.
- **AC-8**: All evidence records for a given `problem_id` are retrievable together. This supports experiment comparison.
- **AC-9**: No structured log event emitted by the Worker contains the `execution_seed` value.

## 8. Testability

1. **Integration: Complete evidence record set — successful job** — Given a complete successful job execution, verify that all required artifact records (job record, decision record, solver run record) are present and queryable by `job_id` after the Worker ACKs.

2. **Integration: Reproducibility tuple completeness** — Given a completed job, verify that the solver run record contains non-null values for all four reproducibility tuple fields: `problem_id`, `execution_seed`, `backend_id`, `contract_version`.

3. **Integration: Re-delivery idempotency — no duplicates** — Given a job message redelivered before terminal state is reached, verify that after re-execution the evidence record set contains exactly one record of each artifact type per `job_id`. No duplicate records exist.

4. **Integration: Terminal state idempotency — no modification** — Given a job in `Completed` state, simulate redelivery of the job message. Verify that the Worker ACKs the message and no artifact record is modified.

5. **Integration: Transient persistence failure — NACK with requeue** — Given a PostgreSQL failure during solver run record persistence, verify that the Worker NACKs with requeue and the job status remains `Processing` (not `Failed`).

6. **Integration: Quality evaluation failure — Completed without QE** — Given a job where quality evaluation fails with an infrastructure error, verify that the job status transitions to `Completed`, the quality evaluation record has `evaluation_status = Failed`, and no failure record is present.

7. **Integration: Pre-solver failure — failure record present** — Given a job that fails at the `Load` stage (routing problem not found), verify that a failure record is persisted with `failure_stage = Load` and the job status transitions to `Failed`.

8. **Integration: Cross-problem retrieval** — Given three jobs for the same `problem_id`, verify that all three evidence record sets are retrievable by `problem_id` query.

9. **Unit: Upsert semantics — decision record** — Given an existing decision record for `job_id = X` with `decision_id = D1`, write a second decision record for `job_id = X` with `decision_id = D2`. Verify that exactly one decision record exists for `job_id = X` and that it contains `decision_id = D2`.

10. **Unit: `execution_seed` not logged** — Instrument all structured log event emission during a job execution. Verify that no log event contains the `execution_seed` value.

SPEC-005 must be consistent with the requirements stated in FR-18.
