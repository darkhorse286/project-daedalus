# Problem Statement

The routing problem model is the foundational domain type for Project DAEDALUS. It is the shared language between all system components.

Without a canonical routing problem definition:

- The API cannot validate job submissions.
- The Core cannot extract workload features.
- The Scheduler cannot score or select solvers.
- Solvers cannot execute.
- The Evidence Log cannot record decisions.
- The Report Generator cannot describe outcomes.
- The Synthetic Workload Generator (SPEC-002) cannot produce valid inputs.

No other component can be fully specified or implemented until this model is approved.

---

# Business Value

This feature unblocks all downstream feature development. Every component in the MVP consumes or produces a routing problem representation.

The model also enables the project thesis by making problem properties measurable. The Scheduler requires quantifiable features (stop count, vehicle count, capacity utilization, time-window density, time-window tightness) to decide whether a solver is eligible and to justify its selection with evidence.

- API Contract Design

Domain modeling for an optimization system requires precise vocabulary, explicit constraint enumeration, and testable validity rules. This feature demonstrates the ability to define a domain type that is simultaneously an API submission contract, a persistence schema, a solver input, and an evidence artifact.

# Requirements

## Domain Concept

A routing problem in Project DAEDALUS is a description of a fleet-routing task. It specifies a set of stops to be visited by a fleet of vehicles operating from a single depot, the capacity constraints governing each vehicle route, and optionally the time constraints on stop arrival. A routing problem describes what must be solved. It does not prescribe how to solve it. The routing problem is the input to every solver, the subject of every Scheduler decision, and the primary subject of every evidence record. It is the shared language between all components.

---

### FR-1: Problem Identity

**Description:** Every routing problem is assigned a system-generated unique identifier at submission time.

**Acceptance Criteria:**
- The identifier is a UUID.
- The identifier is generated by the API before persistence.
- The identifier is returned in the submission response.
- The identifier is persisted with the routing problem.
- The identifier appears on all log events, trace spans, and evidence records associated with the problem.

---

### FR-2: Fleet Definition

**Description:** A routing problem must define the vehicle fleet available to serve stops. In the MVP, all vehicles in the fleet are identical. Single-dimension capacity is used. Multi-dimensional capacity and heterogeneous fleets are deferred beyond the MVP.

**Acceptance Criteria:**
- Vehicle count must be a positive integer.
- Capacity per vehicle must be a positive integer expressed in units of demand.
- Total fleet capacity is vehicle count multiplied by capacity per vehicle.
- A routing problem with zero vehicles is invalid.
- A routing problem with negative vehicle count or negative capacity per vehicle is invalid.

---

### FR-3: Depot Definition

**Description:** A routing problem must define exactly one depot. The depot is the origin and destination for all vehicle routes. All vehicles depart from the depot and return to it after serving stops. Multi-depot routing is deferred beyond the MVP.

**Acceptance Criteria:**
- The depot is specified as a geographic location (latitude, longitude).
- A routing problem with no depot is invalid.
- A routing problem with more than one depot is invalid.

---

### FR-4: Stop Definition

**Description:** A routing problem must define one or more stops. Each stop represents a service point visited by exactly one vehicle. Stop locations use geographic coordinates. Time window fields are optional per stop. Service duration is optional per stop.

**Acceptance Criteria:**
- Each stop has a non-negative integer identifier unique within the problem.
- Each stop has a geographic location (latitude, longitude).
- Each stop has a demand expressed as a non-negative integer in the same units as vehicle capacity.
- Each stop carries an optional service_duration field (semantics defined in FR-16).
- A routing problem with zero stops is invalid.
- A routing problem with duplicate stop identifiers is invalid.
- A stop with a demand exceeding vehicle capacity in a single-vehicle problem is structurally valid but will produce an infeasible result.

---

### FR-5: Location Representation

**Description:** All locations (depot and stops) are expressed as geographic coordinates (latitude, longitude) in IEEE 754 double precision (64-bit floating-point). The C++ Core computes pairwise distances using the Haversine formula, producing great-circle distances in kilometers. This is a spherical Earth approximation acceptable for MVP synthetic scenarios.

**Acceptance Criteria:**
- Latitude is a double-precision value in the range -90.0 to 90.0 (inclusive).
- Longitude is a double-precision value in the range -180.0 to 180.0 (inclusive).
- A location with latitude outside -90.0 to 90.0 is invalid.
- A location with longitude outside -180.0 to 180.0 is invalid.
- Distance computation uses the Haversine formula. No other distance metric is in scope for the MVP.
- The coordinate system is consistent across depot and all stops.
- Coordinates are stored as double precision (float8) in PostgreSQL.
- Coordinates are represented as 64-bit floating-point in the C++ domain representation.

The Haversine formula produces great-circle distances. It does not account for road networks, terrain, or actual travel paths. This limitation must be stated in evidence reports.

---

### FR-6: Reproducibility Seed

**Description:** A routing problem must carry a random seed. The seed is required for all submissions and is part of the canonical routing problem identity. For synthetic problems, the seed records the generator configuration that produced the problem, enabling exact reproduction. For manually submitted problems, the seed has no generator semantics but is recorded as provided and remains part of the problem record.

**Acceptance Criteria:**
- The seed is a non-negative 64-bit integer.
- The seed is accepted as input at submission time.
- The seed is persisted with the routing problem.
- The seed is included in the evidence log and HTML report. This criterion is contingent on the evidence log specification being approved and implemented; it will be verified as part of evidence log implementation, not SPEC-001 implementation.
- For synthetic problems generated by SPEC-002: given the same seed and the same generator configuration, the identical routing problem is produced.
- For manually submitted problems: the seed is accepted as provided without validation of generator intent. It is recorded as-is. For manually submitted problems, the seed does not enable reproducibility unless the caller controls the generation process externally.

The problem seed does not control solver execution directly. The Worker derives an `execution_seed` from the problem seed and provides it to each solver via the SolverRequest (SPEC-004 FR-2). Classical deterministic solvers do not use `execution_seed` for computation; stochastic solvers use it as their exclusive entropy source (ADR-010 Decision 4). The seed records the origin of the problem and enables scenario reproducibility for synthetic workloads.

---

### FR-7: Problem Size Classification

**Description:** A routing problem is classified into one of three size classes based on stop count: Small, Medium, or Large. The classification is a workload feature used by the Scheduler to determine solver eligibility. The routing problem model does not enforce solver-specific size limits. Solver eligibility based on size class is the Scheduler's responsibility.

**Acceptance Criteria:**
- The size class is derived from stop count using configurable threshold values.
- The three classes are: Small, Medium, Large.
- Initial threshold values: Small = 1 to 25 stops, Medium = 26 to 75 stops, Large = 76 or more stops.
- Threshold values are configuration. No code change is required to adjust them.
- The size class is computed by the Core during workload feature extraction, not stored as a field on the routing problem.
- A routing problem is not rejected at submission solely because its stop count falls into a given size class.

The initial threshold values are provisional. They may be revised after QUBO simulated annealing benchmarking establishes the practical feasibility boundary for large problems. Revised values require only a configuration change. These threshold values are also a blocking input for SPEC-003 (Scheduler Objectives and Policy Engine), which cannot define size-based solver eligibility without numeric boundaries.

---

### FR-8: Capacity Feasibility

**Description:** A routing problem where total stop demand exceeds total fleet capacity is infeasible by construction and must be rejected at submission time.

**Acceptance Criteria:**
- Total demand is the sum of all stop demands.
- Total fleet capacity is vehicle count multiplied by capacity per vehicle.
- A routing problem where total demand exceeds total fleet capacity is invalid.
- The rejection response states that the problem is infeasible due to insufficient fleet capacity.

This is a necessary but not sufficient condition for feasibility. A routing problem may pass this check and still be unsolvable due to constraint interactions (for example, time window conflicts). Infeasibility beyond this check is detected during solver execution, not at submission.

Domain validation authority is defined in ADR-009. The API validates for fast caller rejection. The Core re-validates as the canonical authority before solver execution.

---

### FR-9: Constraint Types and Time Window Semantics

**Description:** A routing problem declares which constraint types are active. The active constraint set determines which solvers are eligible. In the MVP, the following constraint types are supported.

**Route Start Reference**

Route start is t=0, defined as the moment vehicles are available to depart the depot. Time window values are non-negative integers representing seconds elapsed from route start. This is a problem-relative time reference requiring no external clock.

**Capacity (required)**

Every routing problem enforces vehicle capacity constraints. Total stop demand must not exceed total fleet capacity (FR-8). Each vehicle route must not exceed vehicle capacity.

**Time Windows (optional)**

A stop may carry a time window defining the interval within which a vehicle must arrive.

Time window fields per stop:
- time_window_open: non-negative integer. Earliest acceptable arrival time, in seconds from route start.
- time_window_close: non-negative integer. Latest acceptable arrival time, in seconds from route start.

If a stop carries time window fields, both time_window_open and time_window_close must be present. A stop with only one of the two fields is invalid.

time_window_open must be strictly less than time_window_close.

Stops without time window fields have no arrival time restriction.

**Early Arrival Behavior**

Early arrival at a time-windowed stop is permitted. If a vehicle arrives before time_window_open, it waits at the stop until time_window_open. Early arrival is not a constraint violation.

Departure time from a time-windowed stop: max(arrival_time, time_window_open) + service_duration.

Departure time from an unbounded stop (no time window): arrival_time + service_duration.

Complete service duration semantics and acceptance criteria are defined in FR-16.

Late arrival (after time_window_close) is a constraint violation. A route where any vehicle arrives after time_window_close at a time-windowed stop is infeasible.

**Constraint Type Flag**

The time_windows constraint type flag is set automatically when one or more stops carry time window fields. A problem with no time-windowed stops does not activate the time_windows constraint type.

**Acceptance Criteria:**
- A routing problem always has the capacity constraint type active.
- A stop with time_window_open greater than or equal to time_window_close is invalid.
- A stop with time_window_open present but time_window_close absent is invalid.
- A stop with time_window_close present but time_window_open absent is invalid.
- The time_windows constraint type flag is true when one or more stops have time window fields, and false otherwise.
- A problem with all stops having time windows is valid provided all window pairs are internally consistent.
- A vehicle arriving before time_window_open waits at the stop until time_window_open. This is not a constraint violation.
- A vehicle arriving after time_window_close produces an infeasible route.

### FR-10: Raw Problem Properties

**Description:** The routing problem must expose raw properties from which workload features can be computed. The Core accesses these properties directly from the routing problem representation.

**Acceptance Criteria:**

The following raw properties must be accessible from the routing problem without additional input:

| Property | Type | Description |
| -------- | ---- | ----------- |
| stop_count | Integer | Number of stops |
| vehicle_count | Integer | Number of vehicles |
| vehicle_capacity | Integer | Capacity per vehicle |
| stop_demands | Sequence | Individual demand value for each stop |
| constraint_types | Set | Active constraint type identifiers: capacity is always present; time_windows is present when any stop carries time window fields |
| stop_time_windows | Sequence | Per-stop time window fields (open, close) where present |
| stop_service_durations | Sequence | Per-stop service duration values, defaulting to 0 when absent |

The derived workload features required by the Scheduler — including capacity_utilization_ratio, problem_size_class, time_window_density, and average_time_window_width_seconds — are computed from these raw properties by the Core. Their computation contract is defined in SPEC-003. SPEC-001 ensures the raw data is accessible; SPEC-003 specifies the feature computation and naming contract.

---

### FR-11: JSON Serialization Contract

**Description:** A routing problem must be representable as a JSON document with a defined schema. This schema is the HTTP API submission format.

**Acceptance Criteria:**
- The JSON schema defines all field names, types, and constraints.
- A document satisfying the schema passes structural validation before domain validation runs.
- A document that does not satisfy the schema is rejected with a structural error before domain validation.
- The JSON document contains sufficient information to reconstruct the full routing problem.
- The schema is version-stable for the MVP.

The Python adapter transport format is explicitly excluded from this spec. It is deferred per ADR-005.

---

### FR-12: Persistence

**Description:** A routing problem must be persisted to PostgreSQL before the associated job is published to RabbitMQ.

**Acceptance Criteria:**
- Persistence and job creation are atomic. If persistence fails, the job is not published.
- A submitted problem is queryable from PostgreSQL using its problem_id.
- The persisted form contains all fields required to reconstruct the C++ domain representation.

---

### FR-13: Scheduler Configuration Association

**Description:** At job submission, a routing problem is associated with a scheduler configuration. The two are separate entities. The routing problem describes what to solve. The scheduler configuration describes how to select a solver.

**Acceptance Criteria:**
- Job submission accepts an optional scheduler configuration identifier.
- When no scheduler_config_id is provided, the default scheduler configuration is used. The default scheduler configuration is defined in SPEC-003.
- The association between problem_id and scheduler_config_id is persisted.
- The routing problem model does not contain solver selection logic or objective weights.
- A scheduler_config_id that does not reference an existing scheduler configuration is a validation error. The submission is rejected with HTTP 400. The system does not silently fall back to the default configuration when an explicit but invalid identifier is provided.

---

### FR-14: C++ Domain Representation

**Description:** The C++ Core constructs an authoritative routing problem representation from the data loaded from PostgreSQL. The C++ representation is the form passed to solvers.

**Acceptance Criteria:**
- The C++ representation contains all fields required by solver execution: geographic coordinates, time window fields, service duration fields, and capacity fields.
- A C++ representation constructed from persisted data is semantically equivalent to the original submission.
- The C++ representation is not the JSON form. The JSON form is the API and persistence form.
- The solver contract defined in ADR-008 operates on the C++ representation.

---

### FR-15: Solver Contract Compatibility (Dependent Architectural Constraint)

**Description:** The routing problem must be representable in a form compatible with the normalized solver contract established in ADR-008.

This is a dependent architectural constraint. It cannot be verified until the concrete solver contract interface is defined during implementation planning. It does not have testable acceptance criteria at this stage.

**Constraint:**
- The routing problem provides inputs for all fields required by the solver contract.
- The concrete solver contract interface is a dependent output of this spec and will be defined during implementation planning following approval.
- No solver bypasses the routing problem representation to access raw submission inputs directly.

---

### FR-16: Service Duration

**Description:** Each stop carries a service duration specifying the time in seconds the vehicle spends at the stop before departing. Service duration applies to all stops regardless of whether time window fields are present.

**Acceptance Criteria:**
- service_duration is an optional field on each stop.
- When service_duration is absent, it defaults to 0 seconds.
- service_duration is a non-negative integer expressed in seconds.
- A negative service_duration value is invalid.
- For a stop with time window fields: vehicle departure time is max(arrival_time, time_window_open) + service_duration.
- For a stop without time window fields: vehicle departure time is arrival_time + service_duration.
- A service_duration of 0 is equivalent to instantaneous service. The vehicle departs at max(arrival_time, time_window_open) for time-windowed stops and at arrival_time for unbounded stops.
- service_duration is persisted with each stop record.
- service_duration is included in the C++ domain representation.
- service_duration is included in the evidence log representation of the routing problem. This criterion is contingent on the evidence log specification being approved and implemented; it will be verified as part of evidence log implementation, not SPEC-001 implementation.

# Non-Requirements

- No real geographic routing. Road networks, traffic data, and turn restrictions are not modeled. Distances are Haversine great-circle approximations only.
- No multi-depot routing. Single depot only. Multi-depot is deferred beyond the MVP.
- No heterogeneous vehicle fleets. Single-dimension capacity only. Multi-dimensional capacity is deferred beyond the MVP.
- No dynamic stop insertion or removal after job submission. The routing problem is immutable.
- No routing problem versioning. A submitted problem cannot be amended.
- No routing problem deletion or archiving. A routing problem is retained indefinitely after submission. No deletion, expiry, or archiving mechanism exists in the MVP.
- No resubmission of an existing routing problem. Each submission creates a new problem with a new problem_id.
- No multi-tenant isolation of routing problems. Authorization is deferred per the README.
- No driver assignment constraints or vehicle-stop affinity preferences.
- No split deliveries. A stop is served by exactly one vehicle.
- No solver-specific size rejection at the routing problem layer. Backend eligibility based on problem size belongs to the Scheduler.
- No real-time coordinate updates during solver execution.
- No Python adapter transport format. This is deferred per ADR-005.

---

# Assumptions

- All vehicles in the fleet have identical capacity. Vehicle heterogeneity is not required for the MVP.
- A single depot serves the entire fleet.
- Stop demand is a single non-negative integer. One-dimensional capacity is sufficient for the MVP.
- Distances are computed using the Haversine formula from geographic coordinates. A precomputed distance matrix is not required as input.
- The routing problem is immutable after submission.
- A routing problem is created once and associated with exactly one job. The problem_id is the foreign key on the jobs table. No routing problem is shared between jobs.
- Routing problems in the MVP are primarily synthetic (generated by SPEC-002). Real-world coordinate submissions are architecturally supported but are not tested in the MVP.
- The routing problem does not contain personally identifiable information in the MVP. Coordinates are synthetic.
- Route start is t=0, defined as the moment vehicles are available to depart the depot. Time window values are seconds elapsed from this reference point. This definition applies to all problems, synthetic and manually submitted.

# Constraints

- The API layer must be C# ASP.NET Core (ADR-002).
- The C++ Core owns the authoritative runtime representation (ADR-001).
- Persistence is in PostgreSQL (ADR-004).
- The solver contract principle defined in ADR-008 governs how the routing problem is consumed by solvers.
- The Python adapter transport format is outside this spec (ADR-005 deferred).
- Problem size classification thresholds must be adjustable without code changes.
- No solver-specific problem size limits may be encoded into the routing problem model.
- Domain validation authority is governed by ADR-009. The API validates for fast caller feedback. The Core validates as the canonical authority before solver execution.
- All geographic coordinates must be represented in IEEE 754 double precision (64-bit) across API deserialization, PostgreSQL persistence, and the C++ runtime representation.

# Failure Modes

### Validation Failure

Cause: Missing required fields, values out of valid ranges, total demand exceeds total fleet capacity, duplicate stop identifiers, invalid time window pairs, invalid coordinates, negative service_duration.

Expected behavior: HTTP 400 response with structured field-level errors. The routing problem is not persisted. No job is published to the queue.

Expected fallback: None. The caller must correct the submission.

User-visible result: HTTP 400 with structured error body.

---

### Invalid Scheduler Configuration Reference

Cause: The submitted scheduler_config_id does not reference an existing scheduler configuration.

Expected behavior: HTTP 400 response with a structured validation error identifying the scheduler_config_id field and stating that the referenced configuration does not exist. The routing problem is not persisted. No job is published.

Expected fallback: The system does not silently fall back to the default configuration when an explicit but invalid identifier is provided.

User-visible result: HTTP 400 with structured error body.

---

### Structural Parse Failure

Cause: HTTP request body is not valid JSON, or a required field has the wrong type.

Expected behavior: HTTP 400 response before domain validation logic executes.

Expected fallback: None.

User-visible result: HTTP 400 with parse or structural error message.

---

### Persistence Failure

Cause: PostgreSQL is unavailable or a write fails.

Expected behavior: HTTP 503 response. No job is published. No partial state is left in the database.

Expected fallback: None. The caller must retry when the service recovers.

User-visible result: HTTP 503 with error message.

---

### Problem Load Failure in Worker

Cause: PostgreSQL is unavailable when the Worker loads the problem, or the persisted record is corrupt.

Expected behavior: The job is not processed. The job moves to the dead-letter queue. A structured failure log event is emitted. The problem.load span is recorded with a failure status. The job status is updated to failed.

Expected fallback: None. Manual investigation is required.

User-visible result: Job status endpoint returns a failed status.

---

### Core Validation Rejection

Cause: The Core rejects a routing problem on load that the API accepted at submission. This indicates a divergence between API validation logic and Core authoritative validation (see ADR-009).

Expected behavior: The job fails before solver execution. A structured validation error is recorded in the evidence log identifying the failed rule. The problem.load span records the rejection. The job status is updated to failed.

Expected fallback: None. The divergence is a detectable fault condition requiring investigation.

User-visible result: Job status endpoint returns a failed status with a validation error code.

---

### Infeasible Problem (Post-Submission)

Cause: A problem passes submission validation but is unsolvable due to constraint interactions detectable only during execution (for example, time windows that cannot all be satisfied simultaneously).

Expected behavior: The solver returns an infeasible result. This is not a submission error. The evidence log records the infeasible outcome. The report describes the infeasibility and the solver's response.

Expected fallback: None. The caller must resubmit with a modified problem.

User-visible result: Job completes with an infeasible result status.

# Testability

The following behaviors must be proven before this feature is considered complete. Specific test names and implementation structure are determined during implementation planning.

## API Validation Behaviors

**Acceptance behaviors — the following inputs must be accepted without error:**

- A routing problem with all required fields, valid coordinates, consistent time windows on all stops, and feasible capacity.
- A routing problem with no time window fields on any stop.
- A routing problem with time window fields on a subset of stops.
- A routing problem where total stop demand exactly equals total fleet capacity.

**Rejection behaviors — the following inputs must be rejected with a structured field-level error identifying the violated field and rule:**

- Zero vehicles in the fleet.
- Zero stops.
- A negative vehicle count.
- A negative capacity per vehicle.
- A negative stop demand.
- Duplicate stop identifiers within the same problem.
- Total stop demand exceeding total fleet capacity.
- A latitude value outside -90.0 to 90.0.
- A longitude value outside -180.0 to 180.0.
- A time window where time_window_open equals time_window_close (degenerate window).
- A time window where time_window_open is greater than time_window_close (inverted window).
- A stop with time_window_open present and time_window_close absent.
- A stop with time_window_close present and time_window_open absent.
- A negative service_duration value.
- A scheduler_config_id that does not reference an existing scheduler configuration.

**Configurability:**

- Problem size class thresholds can be changed through configuration without modifying code. Changed threshold values produce correct size classifications at, above, and below the new boundary values.

## C++ Core Behaviors

**Raw property extraction:**

- stop_count extracted from a loaded problem matches the number of stops in the submitted problem.
- vehicle_count extracted from a loaded problem matches the submitted fleet definition.
- The constraint_types set includes capacity for all problems.
- The constraint_types set includes time_windows for any problem where at least one stop carries time window fields.
- The constraint_types set does not include time_windows for a problem where no stop carries time window fields.

**Distance computation:**

- The Haversine distance between two geographic locations at a known separation, verified against a geographic reference, is within acceptable tolerance for the Haversine formula. This verifies accuracy, not symmetry.

**Time window and service duration:**

- A vehicle arriving at a stop before time_window_open waits at the stop. Departure is at time_window_open + service_duration.
- A vehicle arriving at a stop at or after time_window_open begins service immediately. Departure is at arrival_time + service_duration.
- When service_duration is absent from the stop definition, it defaults to 0 seconds.
- When service_duration is 0, departure time equals max(arrival_time, time_window_open) for time-windowed stops and arrival_time for unbounded stops.

**Persistence round-trip:**

- A C++ routing problem representation constructed from a PostgreSQL record is semantically equivalent to the original submitted problem.
- Geographic coordinates (depot and stops) survive the persistence and load cycle without precision loss.
- Time window fields, service duration values, and capacity fields survive the persistence and load cycle correctly.
- The seed value survives the persistence and load cycle without modification.

## API Integration Behaviors

- A valid problem submission returns HTTP 202 with both job_id and problem_id.
- The routing problem record is persisted to PostgreSQL before the HTTP response is returned.
- A job message is published to RabbitMQ after successful persistence.
- A validation failure returns HTTP 400 with a structured error body identifying the invalid field and the violated rule.
- A malformed JSON body returns HTTP 400 before domain validation logic executes.
- A problem with time window fields has the time_windows constraint type flag persisted correctly.
- A PostgreSQL failure during persistence returns HTTP 503 without publishing a job message.
- A submission with no scheduler_config_id provided returns HTTP 202. The persisted scheduler_config_id is the system default.
- A submission with a scheduler_config_id present but unrecognized returns HTTP 400 with a structured error. The system does not fall back to the default configuration.

## Worker Load Behaviors

- A problem loaded from PostgreSQL by the Worker produces a C++ representation semantically equivalent to the original submission.
- An attempt to load a non-existent problem produces a structured failure log event and routes the job to the dead-letter queue without silent data loss.

## Core Validation Behaviors (ADR-009)

- A routing problem accepted by the API but rejected by Core validation upon load produces a job failure with a structured validation error recorded in the evidence log. The failure is observable through the job status endpoint.

## Observability Behaviors

- A successful submission produces a structured log event including job identifier, problem identifier, stop count, vehicle count, seed, and active constraint types.
- A validation rejection produces a structured log event including rejection context. Raw coordinate arrays and the full stop list must not appear in this event.
- Each problem load by the Worker produces a trace span that records the load outcome and is correlatable with the originating job submission trace.
- Submission volume, validation rejection rates disaggregated by rejection reason, and the distribution of stop counts across submitted problems are observable through metrics.

# Observability Requirements

Observability must be designed alongside this feature.

## Structured Log Events

**On successful submission and persistence**

The API must emit a structured log event when a routing problem is accepted and persisted. The event must carry sufficient information to reconstruct the scope and origin of the submitted problem.

Required information: job identifier, problem identifier, stop count, vehicle count, seed, and active constraint types.

Raw coordinate arrays and the full stop list must not appear in log output.

---

**On validation failure**

The API must emit a structured log event when a routing problem is rejected. The event must carry sufficient context to identify the rejection cause without reproducing the full submitted input.

Required information: stop count (if parseable from the request), vehicle count (if parseable), and a structured list of rejection reasons identifying the violated fields and rules.

Raw coordinate arrays and the full stop list must not appear in this event.

---

**On successful problem load by the Worker**

The Worker must emit a structured log event when the C++ domain representation is successfully constructed from the persisted record.

Required information: job identifier, problem identifier, stop count, vehicle count, seed.

---

**On problem load failure**

The Worker must emit a structured log event when the problem cannot be loaded from PostgreSQL.

Required information: job identifier, problem identifier, failure reason.

---

## Metrics

The following quantities must be observable through metrics. Label cardinality must be managed to prevent unbounded metric growth.

- Total routing problems submitted successfully (counter).
- Total routing problems rejected at validation, disaggregated by rejection reason (counter). The set of rejection reason values must be bounded.
- Distribution of stop counts across submitted problems (histogram).
- Distribution of vehicle counts across submitted problems (histogram).
- Distribution of capacity utilization ratios across submitted problems (histogram).
- Distribution of time window density values across submitted problems (histogram).

## Trace Spans

**job.submit** (existing, in API)

Must carry problem identifier, stop count, vehicle count, seed, and whether time windows are present. Enables correlation and filtering across jobs.

---

**problem.validate** (proposed child of job.submit, in API)

Must record whether validation passed or failed. On failure, must record the count of validation errors. This span is proposed as a child of job.submit because validation is the primary work performed at submission time. If validation is treated as an inline step rather than a discrete span, this decision must be documented during implementation. problem.validate is not in the architecture document's required span list.

---

**problem.load** (in Worker)

Covers loading the routing problem from PostgreSQL and constructing the C++ domain representation. Must record the load outcome (success or failure) and carry job identifier, problem identifier, and stop count. Enables correlation with the originating job.submit span.

# Security Considerations

The routing problem is submitted through the HTTP API, which is a trust boundary. All input must be treated as hostile.

**Input validation**

All fields must be validated against type, range, and domain constraints before any processing occurs. Validation must run before persistence. Coordinate bounds, time window consistency, service duration bounds, and capacity feasibility must all be checked. Invalid inputs must be rejected with structured errors.

**Data sensitivity**

In the MVP, routing problems contain synthetic coordinates and integer demands. No personally identifiable information is expected. If real-world location data is submitted in the future, a privacy review is required before that capability is released. Geographic coordinates representing real locations are sensitive.

**Log safety**

Raw coordinate arrays and full stop lists must not appear in log output. The observability requirements define which information is safe to log.

**Routing problem immutability**

A submitted routing problem cannot be modified. This eliminates a class of consistency vulnerabilities.

**Authentication and authorization**

No authentication or authorization is implemented in the MVP. This is accepted risk for the MVP scope, which targets developer demonstration only.

---

# Performance Considerations

**Validation latency**

Validation must complete quickly enough to give a responsive API. Coordinate validation and time window consistency checks are O(N) in the number of stops. Capacity feasibility is O(N). No specific latency target is established at this stage. Validation performance must be measured during implementation.

**Haversine distance matrix computation**

The C++ Core computes pairwise Haversine distances from geographic coordinates when loading a problem. For N stops plus one depot, this requires N+1 choose 2 distance computations, which is O(N^2). For large problems this computation must be measured and its contribution to total job latency must be reported.

**QUBO matrix size**

QUBO formulations for routing grow quadratically in the number of stops. The initial size classification thresholds (Small 1-25, Medium 26-75, Large 76+) must be validated against QUBO feasibility during benchmarking. Configuration values may be revised based on measurement.

Do not invent specific latency targets. These areas require measurement during implementation.

# Open Questions

## Resolved

**OQ-1 (Resolved): Time Unit Definition for Time Windows**

Decision: Time window values are non-negative integer seconds from route start. Route start is t=0, defined as the moment vehicles are available to depart the depot. This is a problem-relative time reference requiring no external clock. Defined in FR-9.

---

**OQ-2 (Resolved): Service Duration per Stop**

Decision: service_duration is a formal field in the routing problem model. It is an optional non-negative integer field measured in seconds. When absent, it defaults to 0. Departure time semantics are defined in FR-9 and FR-16.

---

**OQ-3 (Resolved): Size Classification Threshold Values**

Decision: Small = 1 to 25 stops, Medium = 26 to 75 stops, Large = 76 or more stops. Thresholds are configurable configuration values. Values may be revised after QUBO simulated annealing benchmarking.

Downstream impact: OQ-3 was a blocking dependency for SPEC-003. Now that threshold values are established, SPEC-003 authoring may proceed.

---

## Open (Non-Blocking)

**OQ-4 (Non-Blocking): Stop and Problem Names**

Should stops carry an optional name field, and should the routing problem itself carry an optional name field?

Names are not used by solvers or the Scheduler. They would appear only in evidence reports and the HTML report generator. Including them adds minimal schema and persistence complexity but improves human readability.

Resolution deferred to implementation planning.

---

**OQ-5 (Non-Blocking): Vehicle Identifiers**

Should individual vehicles carry names or identifiers in the fleet definition?

Vehicle identifiers are not required for optimization. They would appear in reports describing which vehicle serves which stops. The MVP may not produce vehicle-level routing output without additional output specification.

Resolution deferred to implementation planning.
